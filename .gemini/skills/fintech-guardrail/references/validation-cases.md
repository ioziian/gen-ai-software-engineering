# Banking Validation Edge Cases

## 💰 Transaction Amount & Precision
- **Negative/Zero Amount**: Must be rejected with `400 Bad Request`.
- **Floating Point Drift (IEEE 754)**: Ensure `0.1 + 0.2` doesn't become `0.30000000000000004`. Force usage of `Decimal` or `BigInt` (cents).
- **Fractional Cent**: Transactions with >2 decimal places (e.g., `100.555`) must be rejected or rounded according to ISO 20022.
- **Extreme Values**: Protect against `Infinity`, `NaN`, or amounts exceeding `Number.MAX_SAFE_INTEGER`.

## 🔄 Transfer Logic & State
- **Self-Transfers**: `fromAccount === toAccount` must be blocked to prevent artificial volume inflation.
- **Insufficient Balance**: Account balance cannot drop below zero unless a specific "Overdraft" flag is set.
- **Account Status**: Reject transactions if an account is `FROZEN`, `CLOSED`, or `SUSPENDED`.
- **Currency Mismatch**: In a `transfer`, if `currency` is USD but one account is EUR, ensure an exchange rate service is called or reject.

## 🛡️ Concurrency & Integrity
- **Double Spend (Race Condition)**: Two simultaneous withdrawals exceeding the balance must not result in a negative balance. Requires atomic updates or row-level locking.
- **Idempotency**: Retrying a timed-out request with the same `X-Idempotency-Key` must return the original result, not create a duplicate transaction.
- **System Time Drift**: Ensure `timestamp` is generated by the server/database, not trust the client clock.

## 🆔 Identifiers & Metadata
- **ID Format Injection**: Ensure `accountId` strings are sanitized to prevent path traversal or SQL injection.
- **Duplicate Transaction ID**: Hard rejection on storage level to prevent audit log corruption.
